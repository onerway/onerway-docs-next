# Markdown 文档综合审查

请基于 `content/CLAUDE.MD` 文档规范指南，对指定的 Markdown 文件进行全面深度审查。

**目标文件**: $ARGUMENTS

如果未提供文件路径，请审查最近修改的 Markdown 文件。

---

## 📋 审查维度（基于 Onerway 文档规范）

### 一、易于浏览性

#### 1.1 内容结构组织
- **章节标题**
  - [ ] 标题是否信息丰富？（使用完整句子而非抽象名词）
  - [ ] 标题层级是否正确？（H1 → H2 → H3 → H4，不跳级）
  - [ ] H1 标题是否唯一？
  - [ ] 读者能否通过标题快速判断章节内容？

- **段落结构**
  - [ ] 段落是否简短易读？（避免超长段落）
  - [ ] 重要观点是否单独成段？
  - [ ] 段落首句能否独立理解？（不依赖上下文）
  - [ ] 主题词是否放在句首？
  - [ ] 重要信息是否前置？（结果优先于过程）

#### 1.2 格式化使用
- **列表和表格**
  - [ ] 是否适当使用列表展示并列内容？
  - [ ] 表格格式是否正确且易读？
  - [ ] 列表格式是否统一？（-、1.、[ ]）

- **强调和突出**
  - [ ] 关键信息是否加粗？（API 密钥、重要提示、必需参数）
  - [ ] 是否避免过度使用强调？

### 二、句子质量

#### 2.1 简洁清晰
- [ ] 句子是否简短直接？（拆分长句）
- [ ] 是否删除了不必要的词汇？
- [ ] 是否多用祈使语气？（`创建订单` 而非 `你需要创建订单`）
- [ ] 句子是否避免歧义？（只有一种解析方式）
- [ ] 是否避免左分支句子？（不让读者记忆太多信息）

#### 2.2 指代清晰
- [ ] 是否减少代词使用？（尤其跨句子的 `这`、`那`、`它`）
- [ ] 代词指代是否明确无误？
- [ ] 是否避免假设读者的想法？（`你可能想...`）

#### 2.3 一致性
- [ ] 术语使用是否统一？（不混用不同说法）
- [ ] 代码示例格式是否一致？
- [ ] 是否避免中英文混用？（保持术语一致）

### 三、读者友好性

#### 3.1 语言简单性
- [ ] 是否使用简单直接的语言？
- [ ] 缩写是否在首次出现时解释？（API、ACH 等）
- [ ] 是否避免过度使用技术术语？

#### 3.2 完整性和实用性
- [ ] 是否预见并解决潜在问题？
- [ ] 示例是否完整可运行？
- [ ] 是否提供多平台示例？（macOS/Linux/Windows）
- [ ] 是否提供背景信息？（宽泛的开场介绍）

#### 3.3 代码示例质量
- [ ] 代码示例是否通用且可导出？
- [ ] 是否最小化依赖？
- [ ] 是否避免硬编码敏感信息？（使用 `sk_test_...` 占位符）
- [ ] 是否包含正确的错误处理？
- [ ] 是否提供多语言示例？（Node.js、Python、PHP 等）
- [ ] 代码块是否指定语言？

### 四、Onerway 项目特定规范

#### 4.1 Frontmatter 验证
- [ ] 是否包含必需的 `title` 字段？
- [ ] 是否包含必需的 `description` 字段？
- [ ] YAML 格式是否正确？
- [ ] `description` 长度是否合理？（50-160 字符，用于 SEO）
- [ ] 字段值是否非空且有意义？

#### 4.2 多语言一致性
- [ ] 是否存在对应的其他语言版本？（en/ ↔ zh_cn/）
- [ ] 文件路径结构是否对应？
- [ ] 标题层级是否一致？
- [ ] 代码示例是否保持一致？
- [ ] 专业术语翻译是否统一？

#### 4.3 MDC 组件语法
- [ ] 组件语法是否正确？（`::component-name ... ::`）
- [ ] 属性写法是否正确？
  - 简单值：`key="value"` 或 `key=true`
  - 数组/对象：必须加 `:` 前缀 → `:items='["A","B"]'`
- [ ] 复杂数据是否使用 YAML 块？（推荐）
- [ ] 嵌套组件是否正确？（使用 `:::` 三冒号）
- [ ] 命名插槽是否正确？（`#header`、`#body`）

**常见组件检查**：
- [ ] `::code-group` 代码组是否正确？
- [ ] `::link-switch` 链接切换器是否使用 preset？
- [ ] `::accordion` / `::accordion-item` 折叠面板是否正确？
- [ ] `::tabs` / `::tab-item` 标签页是否正确？
- [ ] `::note` / `::warning` 提示框是否适当使用？

#### 4.4 链接规范
- **内部链接**
  - [ ] 是否使用相对路径？
  - [ ] 是否避免包含语言前缀？（系统自动处理）
  - [ ] 链接路径是否正确？

- **外部链接**
  - [ ] 是否使用 HTTPS？
  - [ ] 链接是否有效？
  - [ ] 链接描述是否有意义？（避免"点击这里"）

#### 4.5 图片和媒体
- [ ] 图片路径是否正确？
- [ ] 是否包含 alt 文本？
- [ ] 图片格式是否优化？

### 五、SEO 和可访问性
- [ ] description 长度是否符合 SEO 最佳实践？（50-160 字符）
- [ ] 是否使用语义化 HTML 标记？
- [ ] 标题层级是否符合可访问性标准？

## 📊 输出格式

请按以下结构提供详细审查报告：

### ✅ 优点（文档做得好的地方）
列出文档的亮点，具体说明哪些方面符合规范。

### ❌ 关键问题（必须修复）
列出严重的格式、内容或规范违反问题，包括：
- 具体问题描述
- 所在位置（行号或标题）
- 建议的修复方案

### ⚠️ 警告（建议修复）
列出可以改进的地方，包括：
- 改进点描述
- 当前写法示例
- 推荐写法示例

### 💡 建议（可选改进）
列出进一步优化的建议，如：
- 内容增强建议
- 用户体验提升点
- 组件使用建议

### 🌐 多语言一致性（如适用）
如果文档有对应的其他语言版本，检查：
- 结构是否对应
- 是否有缺失或多余的章节
- 代码示例是否一致

---

## 📌 审查提示

1. **全面性**：使用 Read 工具读取完整文件内容
2. **对比性**：如有对应的其他语言版本，使用 Read 工具对比检查
3. **规范性**：严格参照 `content/CLAUDE.MD` 中的规范标准
4. **实用性**：提供具体可操作的改进建议
5. **优先级**：按严重程度排序问题（关键 > 警告 > 建议）

开始审查吧！
