# Set Payment Methods

## How to configure a range of global payment methods.

利用 `lpmsInfo`，可通过单一 API 接受多种支付方式。`lpmsInfo` 对象包含创建付款所需的支付方式详情。利用 `lpmsInfo`，您可以组合 `productType`：

- 使用聚合支付，在 Onerway 托管页面上展示所有已启用的支付方式供用户选择

```json
"productType": "ALL"
```

- 用 `lpmsInfo` 和 `productType: "ALL"` 配合使用，可以指定托管页面上只展示某一个支付方式

```json
"productType": "ALL",
"lpmsInfo": "{\"lpmsType\":\"Skrill\"}"
```

## Supported Payment Methods

要确定某个具体地点使用哪种支付方式，请查看支付方式指南。

该指南包括不同地区可以使用的支付方式、每种支付方式特征的详细说明以及它们最相关的地理区域。您可以在管理平台中启用任何可用的支付方式。激活通常是即时的，不需要额外的合同。

## Customer Actions

某些支付方式要求您的客户采取不同的操作来完成付款。API 响应参数中通过 `actionType` 参数指定客户操作的类型。

客户需要采取的常见操作包括：

- **重定向至银行页面验证**：重定向到客户银行的线上服务页面以验证并批准付款。（`actionType: RedirectURL`）
- **扫码支付**：向客户展示二维码，客户通过扫码完成付款。（`actionType: QrCode`）
- **调起钱包支付**：根据响应参数中的 `codeForm` 和 `presentContext` 值，请求对应钱包的 SDK，在付款过程中主动拉起客户移动端的钱包进行支付（例如微信支付）。（`actionType: ShowContext`）

::callout{type="info" title="注意"}
信用卡支付通常不需要额外的客户操作（要求 3DS 验证的除外）。3DS 验证要求在收集银行卡详情后进行进一步身份验证。
::

## Immediate or Delayed Payment Notification

有些支付方式会在尝试交易后立即返回付款状态（例如银行卡付款），但其他方式会有一个延迟，例如 Trustly。

对于即时确认的支付方式，您可以通过以下两种方式确认收款成功（`status: S`）：

- 接收 Webhook 通知
- 客户重定向到您提供的 `returnUrl` 后，通过轮询查询接口确认状态

延迟通知型支付方式在延迟期间不保证支付成功，status 会是 `P`，直至付款状态成功或是失败。这段时间内，很多商家会保持处于 _处理中_ 状态的订单，等待付款成功后再履行订单。

::callout{type="warning" title="注意"}
对于延迟确认型支付方式，请务必配置 Webhook 端点以接收付款状态的最终通知。
::

## Single-use or Reusable

您可以重复使用某些支付方式（例如银行卡或银行借记）进行额外的付款，不需要再次授权和收集支付详情。

为减少未来的银行拒付和支付阻力（例如[要求的验证](https://docs.stripe.com/strong-customer-authentication)），您应该始终设置可重复使用的支付方式以供未来使用。对于可重复使用的支付方式，可以在接受付款时设置以供未来使用，也可以在不接受付款的情况下设置以供将来使用。

单次使用的支付方式（例如某些类型的银行转账）不能绑定到客户，因为在尝试付款后即作废。
